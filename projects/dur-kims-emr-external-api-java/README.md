# DUR / 개인투약이력 EMR 외부조회 API (Java · EMR 컴포넌트)

> ⚠️ 공개본 안내: 실제 [의료기관] EMR 서브밋ID·컴포넌트 경로·서버 URL·프로시저/테이블명·주민번호는 포함하지 않습니다. 구조만 일반화했습니다.

## 문제
EMR 처방화면의 DUR(의약품 안전점검)과 타기관 처방이력(개인투약이력)이 화면(xfdl) 안에만 존재해, n8n·타 시스템에서 재사용할 수 없었다.

## 접근
화면 기능을 **재사용 가능한 내부 API로 개방**한다. 기존 스크리닝 서비스를 오케스트레이션하는 전용 컴포넌트를 신규 개발하고, 개인투약이력은 기존 인터페이스 컴포넌트에 메서드/DAO/SQL/맵을 추가한다.

## 아키텍처 (레거시 EMR 컴포넌트 계층)
```
외부 호출(POST /*.jnu, jparam)
  → app(외부노출 req*)         # 신규 전용 컴포넌트
    → mgr(오케스트레이션)       # 실행 + 조회 서비스 조합
      → dao / procedure        # HIRA 전송·DB 기록
  → 결과 dataset 반환
```
반영 순서: **SQL → DAO → Mgr → App → map** (비대칭 커밋 위험 차단).

## 핵심 기술 / 안전장치
- 기존 스크리닝 서비스 재사용(실행 + 조회 서비스 조합), 입력 12파라미터·출력 커서/상세 매핑
- classpath 문제(`org.json.simple` 부재) → **gson 전환**
- **정적검증 반영**: null 방어, 주민번호 유효성 사전차단, 파라미터 key 중복 제거 헬퍼
- 크리덴셜/주민번호는 **로컬 변수로만, 로그 출력 금지**
- "BUILD SUCCESSFUL ≠ 정합" 원칙 → 실호출 + DB 재조회 검증
- 배포: 본체+config+map+build 한 묶음 커밋 → ant 빌드 → WAS 재기동 → 카나리 1건 → DB 검증

## 사용 기술
Java, iBatis(`*_sqls.xml`), 레거시 EMR 프레임워크(서비스정의/맵/빌드), Oracle/PL-SQL, gson, SVN + ant.

## 역할
EMR Java 컴포넌트 **신규/수정 직접 개발**.

## 보여주는 역량
레거시 EMR 내부(서브밋·컴포넌트·DB)까지 안전하게 신규 API를 얹는 개발 깊이, 정적검증·실검증 병행.

## 가상 재구현
`impl/` — 가상 EMR의 app/mgr/dao 계층과 의약품 상호작용 점검 API 데모(합성 처방/약품 데이터).
